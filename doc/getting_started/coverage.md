# Code Coverage User Guide

This guide provides practical instructions for OpenTitan developers to measure
and analyze code coverage for device-side software.

The OpenTitan coverage system supports C/C++, Ibex ASM, and OTBN Assembly across
multiple execution environments, including host PCs, FPGA, QEMU, and the OTBN
simulator.

## Prerequisite

To run coverage-enabled tests and generate reports, the following software must
be installed on your host system:

-   **`llvm`:** Required for processing unit test coverage mapping and counters.
    Tested with versions 14, 16, and 19.
-   **`clang`:** Required for compiling unit tests compatible with `llvm` tools.
-   **`lcov`:** Required for generating standard coverage data files and HTML
    reports.
-   **`python3`:** Required for running the coverage collection and viewer
    generation scripts.
-   **`rsync`:** Required for the coverage collection and merging scripts.
-   **Common CLI Tools:** Standard utilities such as `grep` and `sed` are used
    by the processing pipeline.

To run the tests, you must use the `bazel coverage` subcommand with the
`--config=ot_coverage` flag. This flag ensures that the correct instrumentation
is applied and that the host capture tools are active.

Coverage is supported on the following platforms:

-   **Host PC:** For host-side unit tests.
-   **FPGA (CW310/CW340):** Real hardware execution with UART-based capture.
-   **QEMU:** Emulated execution with console-based capture.
-   **OTBN Simulator:** For OTBN application coverage.

## Running Tests

Coverage can be measured for various types of tests by adding the coverage
configuration to the `bazel coverage` command. You can run multiple tests in a
single invocation to generate a combined report:

```bash
# Define the set of tests to run with coverage
# On-device tests can be executed on both FPGA on QEMU platforms.
TESTS=(
  # Host-side unit tests
  "//sw/device/lib/base:math_unittest"

  # Ibex C functional tests
  "//sw/device/tests:uart_smoketest_sim_qemu_rom_with_fake_keys"
  "//sw/device/tests:uart_smoketest_fpga_cw340_test_rom"

  # Ibex ASM functional tests
  "//sw/device/tests:crt_test_sim_qemu_rom_with_fake_keys"
  "//sw/device/tests:crt_test_fpga_cw340_test_rom"

  # ROM\_EXT E2E tests
  "//sw/device/silicon_creator/rom_ext/e2e/verified_boot:key_dev_sim_qemu_rom_ext"
  "//sw/device/silicon_creator/rom_ext/e2e/verified_boot:key_dev_fpga_cw340_rom_ext"

  # ROM E2E tests
  "//sw/device/silicon_creator/rom/e2e/address_translation:rom_ext_v_flash_a_sim_qemu_rom_ext"
  "//sw/device/silicon_creator/rom/e2e/address_translation:rom_ext_v_flash_a_fpga_cw340_rom_ext"

  # Provisioning tests
  "//sw/host/provisioning/orchestrator/tests:e2e_emulation_dice_cwt_cw340_test"

  # OTBN application tests (simulator-based)
  "//sw/otbn/crypto/tests:boot_testcase_boot_mode_invalid"

  # Coverage views for firmware analysis (filtered reachable code)
  "//sw/device/silicon_creator/rom_ext/imm_section:imm_section_dice_x509_coverage_view"
  "//sw/device/silicon_creator/rom_ext/sival:rom_ext_prod_dice_x509_xmodem_coverage_view"
)

# Run all selected tests with coverage enabled
./bazelisk.sh coverage --config=ot_coverage --test_output=all "${TESTS[@]}"
```

## Analyzing Reports

After the `bazel coverage` command completes, the results are processed and
stored in the `bazel-out/_coverage` directory.

### LCOV HTML Report

Bazel generates a combined LCOV data file. You can find the raw report at:
`bazel-out/_coverage/_coverage_report.dat`

To generate a human-readable HTML view from the combined results:

```bash
genhtml -o /tmp/$USER/coverage --ignore-errors inconsistent,unsupported \
  bazel-out/_coverage/_coverage_report.dat
```

### OpenTitan Coverage Viewer

For high-assurance analysis and per-test traceability, OpenTitan provides a
specialized web-based Coverage Viewer (see
**[documentation](../../util/coverage/viewer/README.md)**). This is typically
generated by the CI pipeline for nightly builds, but can be created locally
using the tools:

```bash
python3 -m util.coverage.viewer.collect
python3 -m util.coverage.viewer.bundler
```

or using the CI scripts:

```bash
bash ci/scripts/collect-coverage-report.sh
bash ci/scripts/merge-coverage-report.sh
```

## Troubleshooting & Debugging

Measuring coverage on embedded targets can be complex. Use the following tools
and techniques to diagnose issues.

### Debugging Tools

-   **Verbose Mode:** Run Bazel with `--test_env=VERBOSE_COVERAGE=1`. This will
    display detailed error messages during the host-side coverage collection and
    processing stages.
-   **Sandbox Debugging:** Use the `--sandbox_debug` flag with Bazel. This
    preserves the execution root after the test finishes, allowing you to
    inspect intermediate files and verify path issues.

### Common Issues

#### 1. Empty or Missing `.dat` Output

If a test completes but no coverage data is generated:

-   **Check for Collection Errors:** Use `VERBOSE_COVERAGE=1` as described above
    to see if the host-side coverage tools are failing.
-   **Verify Transmission:** Check the test console log. You should see the `==
    COVERAGE PROFILE START ==` and `== COVERAGE PROFILE END ==` anchors. If they
    are missing, the firmware might be crashing before reporting or the report
    call might be missing.
-   **Check Runtime Linking:** Ensure the correct coverage runtime is linked. If
    your firmware is not based on the OTTF (e.g., a custom boot stage), you must
    manually include the UART or OTTF coverage runtime in your dependencies to
    avoid linking the weak no-op implementation.

#### 2. Incorrect "Uncovered" Lines

If code that you know was executed is marked as uncovered:

-   **Profile Check:** Ensure the profile for the target firmware is actually
    listed in the test log.
-   **Reporting Timing:** Verify in the log that the coverage data was reported
    *after* the code in question was executed and *before* any device reset.
-   **Raw Data Audit:** Inspect the raw per-test `.dat` file in the
    `bazel-testlogs` folder to see if the counters were captured.
-   **Assembly Counters:** For RISC-V assembly, verify that a counter was
    assigned to the basic block. You can use the automated instrumentation
    tool's `--dryrun` mode to view the block analysis results.

#### 3. Build ID and Profile Issues

**Missing profile with build id xxxxx:** This error indicates that the coverage
tool received a report from the device but could not find the corresponding ELF
binary on the host to correlate the counters.

-   **Verify Runfiles:** Ensure that the ELF binary is included in the
    `runfiles` of the `opentitan_test` target. OpenTitan's custom Bazel rules
    should propagate these automatically, but manual intervention may be needed
    for complex test structures.
-   **Provisioning Tests:** For provisioning E2E tests, you may need to manually
    add the target firmware binary to the `data` attribute of the test rule to
    ensure it is available in the runfiles.

#### 4. Test Fails Only in Coverage Mode

If a test passes in a normal build but fails when coverage is enabled:

-   **Exec Environment:** If the test supports the `test_rom` execution
    environment, try running it with that first, as it is the most similar to a
    standard build.
-   **Flash Layout:** Check if the test relies on absolute flash addresses.
    Instrumentation shifts the layout; use the `slot_spec` variables to make the
    code layout-agnostic.
-   **Instruction Interference:** Coverage instrumentation adds instructions to
    update counters in SRAM. This can affect low-level code sensitive to
    latency, timing, or specific memory access patterns.

#### 5. Linker and Resource Errors

-   **Memory Exhaustion:** Check the build log for "RAM exhausted" or "Flash
    exhausted" errors. This often manifests as overlapped section errors.
-   **Reduce Footprint:** If a binary is too large, disable coverage for
    uninteresting libraries (e.g., generated `ujson` code or testing DIF
    libraries) by adding `features = ["-coverage"]` to those targets.
-   **SRAM Binaries:** Code designed to run entirely from SRAM often has very
    tight resource constraints. Instrumenting such binaries for coverage is
    frequently infeasible due to the overhead of the counter sections.
